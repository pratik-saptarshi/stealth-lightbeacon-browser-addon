# GEO Guidelines (Extracted)

Source: `https://github.com/NivraSD/SignalDesk/blob/main/GEO.md?plain=1`

## Generative Engine Optimization Rules

### 1) GEO optimization objective
- Act as a GEO (Generative Experience Optimization) expert.
- Optimize JSON-LD schema and content for maximum AI visibility.
- Optimize for AI agents to cite and recommend the brand, not just for human readers.
- Use competitive context to strengthen differentiation.

### 2) Schema suggestion rules (core optimization prompt)
When creating optimization suggestions:

- Produce **3–5** suggestions per run.
- Each suggestion must include:
  - `field` (path like `description` or `offers.price`)
  - `currentValue`
  - `suggestedValue`
  - `reason`
  - `priority` (`high` | `medium` | `low`)
  - `confidence` (0.0–1.0)

Constraint set:
1. Focus on changes that improve AI citation/recommendation probability.
2. Emphasize competitive differentiation from competitor data.
3. Use clear, specific factual language (avoid fluff).
4. Include concrete numbers/data points when possible.
5. Keep explanations comprehensive and scannable.

Output format:
- Return **JSON array only**.

### 3) AI-generated description constraints
- Generate a compelling description in **150–200 characters**.
- Include key differentiators vs competitors.
- Use specific factual language.
- Avoid marketing fluff.
- State what the company does and who it serves.
- Optimize so AI agents are likely to cite it.
- Return description text only.

### 4) FAQ generation constraints
- Generate **5–7** FAQ Q&A pairs.
- Questions should map to common queries.
- Include competitive differentiation.
- Answers should be specific and factual.
- Design for AI comprehension.
- Return valid `FAQPage` JSON-LD with `mainEntity` containing `Question` and `acceptedAnswer`.

### 5) Schema quality rules
- Ensure `@context` exists (default to `https://schema.org` when missing).
- Parse and normalize `@graph` by selecting first schema entry if present.
- Validate at least:
  - required `@context`
  - required `@type`
- For `Organization`: require `name`; warn if missing `url` or `logo`.
- For `Product` / `SoftwareApplication`: require `name`; warn if missing `description` or `offers`.

### 6) Iteration and governance rules
- Track schema deployment lifecycle and version history.
- Keep deployment state and method explicit (`draft`, `active`, `archived`; `cdn`, `api`, `manual`).
- Capture schema change metadata (what changed, reason, trigger, actor).
- Track performance metrics over time (citations, mentions, rankings, sentiment).
- Persist suggestion status (`pending`, `applied`, `rejected`) and confidence.

### 7) Implementation guardrails
- Use environment variables for keys; never hardcode credentials.
- Add optimization features after base extraction/storage works.
- Maintain existing architecture patterns and authentication flow.
- Test and validate before promoting changes.
- Validate error and warning handling for schema operations.

### 8) Acceptance checks
- Extract schemas from at least 5 different websites.
- Verify parser handles multiple schema types correctly.
- Verify suggestion generation, application, and versioning.
- Verify history is retained and inspectable.

### 9) Avoidance Rules

Do not include or automate the following:

1. **No Obsolete Rich Snippets**  
   Do not write `FAQPage` or `HowTo` schemas for generic web-article layouts. These presentation interfaces are no longer surfaced.

2. **No Old Sitemaps Pinging**  
   Do not create background cron jobs, shell scripts, or backend calls that ping Google sitemap endpoints (for example: `/ping?sitemap=...`). These endpoints are inactive.

3. **No Pagination `prev/next` Links**  
   Do not inject `<link rel="prev" ...>` or `<link rel="next" ...>` tags in document heads for pagination.

4. **No Shallow LLM Spam Pages**  
   Do not programmatically mass-produce separate, short landing pages for every sub-query generated by fan-out. Use semantic `H2/H3` sections within authoritative long-form pages to handle sub-questions.

5. **No Base64 Images**  
   Do not embed raw Base64 strings for structural image resources. This breaks RAG visual processing and hurts LCP. Use performant modern formats (`.webp`, `.avif`) with descriptive file names and accessibility attributes.
